# PERF_REPORT — iPhone最適構成のための要素別コスト計測（Phase 1）

作成: 2026-07-12 ／ 対象: seasons-gpu（WebGPU版β）main ／ ハーネス: `scripts/qa-perf.mjs`（`npm run perf`）

## 計測条件

- iPhone模擬: ビューポート393×852・deviceScaleFactor 3・**CPU 4倍スロットル**（CDP）・実ウィンドウ=本物のWebGPU（RTX 4050）
- プロトコル=**ペア比較**: デスクトップはCPUブースト減衰で実行順に遅くなるため、各要素を［基準→要素OFF］の隣接ペアで測り差分をコストとする。基準は `?dpr=1.6` 固定（自動品質を凍結した公平条件）
- 絵への貢献 = ペアのスクショ（CSS層込み）を393×852へ縮小した**平均画素差%**。`noise`行＝同一構成同士の差＝ノイズ床
- シナリオ: **fly**=低空飛行・昼（街一望・描画コール1,900） ／ **window**=窓辺・夜（2,100コール・ブルーム強） ／ **cloud**=雲海高度（半透明の重ね塗り最大）

### この計測でわかること・わからないこと（正直な限界）

- **わかる**: CPU側コスト（描画コール送出・毎フレームJS）の相対比較、絵への貢献度、描画コールの内訳
- **わからない**: デスクトップGPUは桁違いに速いため、**画素側（CSS合成層・DPR・ブルーム・FXAA・影サンプリング）のGPUコストはfpsに現れない**。これらは「塗る画素数×パス数」の解析モデル＋実機の既知事実（過去の実機AB・自動品質が実機で下限まで落ちた実測）で評価する
- window/cloudの一部ペアに2〜2.5倍の外れ値あり（WindowsのP/Eコアスケジューリング起因とみられる二峰性）。外れペアは中央値側を採用して解釈

### 画素仕事の解析モデル（GPU＝発熱側。デスクトップでは測れないため計算で出す）

1フレームあたり（iPhone想定・CSS px 393×852）:

| 項目 | 画素量 | 対WebGL比 |
|---|---|---|
| WebGL本描画（DPR1.6） | 629×1363 ≈ **0.86MP**（＋オーバードロー） | 1.0 |
| FXAA＋出力変換 | ≈0.86MP | +1.0 |
| ブルーム（夜のみ・多段mip） | ≈0.5MP相当 | +0.6 |
| **CSSブレンド4層（ネイティブ解像度で合成）** | **3.0MP×4層 ≈ 12MP** | **≈ +5.5**（WebGL全パイプの5倍超） |

→ 発熱（総電力）の観点では、**WebGLの外側にあるCSS合成がピクセル仕事の最大項**。過去の発熱対策計画でも最大レバーと特定されながらシェーダー内蔵化3回失敗で保留中の項目と一致。

## コスト対効果ランキング

**削減候補（コスト高 × 貢献低〜中）**
1. **CSSブレンド層の絞り込み（4→1〜2層）** — 画素仕事の最大項（12MP/フレーム）に対し、貢献は全体で3.3〜3.6%。ただし「水彩の同一性」は作品の核なので、紙目（multiply）を残す絞り込みが正解。全OFFは不可（スクショで生CG化を確認）
2. **飛行中のDPR帯の見直し（1.6→1.35固定）** — 画素-29%。動きがボケを隠す場面。しかも実機では自動品質が既に1.24-1.36へ下げている（=ユーザーは今も1.6を見られていない）。「固定の中庸帯」にすれば荒れの発振も消える
3. **雲海帯の同見た目メッシュ統合** — 265コール（灯籠18×3メッシュ・光の粒・加算スプライト群）を見た目を変えずに統合可能

**維持（コスト低 × 貢献中〜高、または既に最適化済み）**
- 静的影（焼きは一度だけ・CPUコストゼロ。切る価値なし）
- 夜ブルーム（局所2.1%だが夜の情緒の要。昼は既にOFF）
- FXAA（輪郭線アートでジャギーは目立つ・1パスの価値あり）
- 雲海の絵そのもの（コスト特大だが貢献12%=切ってはならない）

**構造項（今回のスコープ外の大物）**
- 町並み本体の描画コール約1,300-1,900（家・建物）— 統合には建物ジオメトリの再設計が必要（既存バックログ#260）。今回の最適構成の外に置くが、次の大幅改善はここ

## Phase 3 実装ログ（案B・承認済み）

### C1: CSSブレンド層 4→2（eac70df → 改良 602e8e5・A/B用スイッチ 39b1195）
- 初案（紙目のみ残す）は純粋A/B（同一ページ内トグル＝アニメ差ゼロ）で **昼4.0%／夜3.5%**、目視で「夜の暖色トーンが寒色に沈む」を確認→**不採用・作り直し**
- 採用案（wash=情景の光の暖色を残し、paper2/bleedのみ畳む）: **昼1.81%／夜2.67%**、夜の情緒維持を目視確認
- 効果（解析）: コンポジタのネイティブ解像度ブレンド合成 **12MP→6MP/フレーム（-50%）**＝発熱最大項の半減
- CPU実測: 変化なし（ノイズ内・想定どおり）。`?fullcss=1` で旧4層を完全再現可（実機A/B用）

### C2: 飛行中DPR 1.2＋自動品質の下限 0.96→1.2（91192e0）
- 機能実測: **窓辺1.6→飛行1.2→窓へ帰還1.6→着地歩行1.6** の遷移OK・エラー0
- 画素仕事: 飛行中 **-44%**（1.6比）。「画質がすごく悪い」状態（実機FBの0.96張り付き）は構造的に発生不能に＝**最低画質の底上げ（買い戻し）**
- 仕上げ計測: fly auto の実効DPRが**1.2に固定**（Phase 1では0.96〜1.24で発振）

### C3: 能動操作中fps上限 30→40（0013768）
- rAFのvsync量子化により、iPhone Safari既定（60Hz）＝従来どおり安定30fps／ProMotion 120Hz有効環境のみ真の40fps＝**ガタつく擬似40fpsを生まない安全な引き上げ（買い戻し）**。眺め時16fpsの省電力は不変
- 実測: 能動34.8fps（デスクトップ・上限内）・エラー0

### C4: 雲海帯の統合 → **計測により見送り**（1cc971f）
- 内訳実測（__town3dCloudBreak）: 雲海高度347コールの主は**島々・吊り橋・入道雲（277）**。統合候補と見ていた漂いもの（灯籠・光の粒・湯けむり）は視界内でほぼ0、灯籠流し33＋光芒14のみ
- iPhone実測単価（2,100コール≈6.15ms）換算で277コール≈**1ms未満/フレーム**＝発熱への寄与は微小
- 島統合はreveal高度フェード（材質opacity制御）・住人アニメと絡む高リスク工事→**コスト対効果が不成立のため中止**（絵の白眉を守る）。描画コールの本命は窓辺/飛行の約2,100＝建物本体（バックログ#260）

### 仕上げ検証（最終構成）
- 全25情景スモーク: エラー0
- 出荷時auto実測（CPU4xスロットル）: 窓辺夜 calls2100・pr1.36（変動域1.2〜1.6）／低空飛行 calls1951・**pr1.2固定**／雲海 calls354・pr1.2
- 総合（iPhone想定・解析）: ピクセル仕事 CSS 12→6MP ＋ 飛行中WebGL 0.86→0.48MP×パス ≈ **合計-40%前後**。最低画質+56%画素（0.96²→1.2²）
